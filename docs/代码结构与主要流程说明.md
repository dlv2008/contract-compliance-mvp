# 代码结构与主要流程说明

## 1. 当前代码处于什么阶段

当前这套代码不是完整业务系统，而是一个 `Day 3` 原型骨架。

它已经具备：

- `FastAPI` 应用入口
- 健康检查接口
- 两个基础页面：
  - 工作台总览页
  - 审查结果页
- 一份前端样式文件
- 一份模拟数据文件

它暂时还没有接入：

- 真实数据库
- 文件上传
- 任务持久化
- `RAGFlow`
- `MiniMax / Qwen`
- 规则引擎

所以你可以把它理解成：**先把页面形态和最小服务骨架搭出来，再逐步把真实链路接进去。**

## 3. 当前主流程是怎么跑的

当前主流程非常简单，可以按下面这条链理解：

### 3.1 启动应用

从 [app/main.py](C:/Users/dlv20/Documents/合同合规审查/apps/api/app/main.py) 创建 `FastAPI` 应用。

这里做了 4 件事：

1. 创建应用对象
2. 注册 `health_router`
3. 注册 `ui_router`
4. 挂载 `/static` 静态资源

### 3.2 用户访问页面

比如你打开 `/reviews/demo-001` 时：

1. 请求先进入 [ui.py](C:/Users/dlv20/Documents/合同合规审查/apps/api/app/routes/ui.py)
2. 命中 `review()` 函数
3. `review()` 调用 `build_review()`
4. `build_review()` 从 [sample_data.py](C:/Users/dlv20/Documents/合同合规审查/apps/api/app/sample_data.py) 返回一份字典数据
5. 这份数据被传给 `review.html`
6. `Jinja2` 模板把数据渲染成 HTML
7. 浏览器再加载 [styles.css](C:/Users/dlv20/Documents/合同合规审查/apps/api/app/static/styles.css) 完成页面展示

### 3.3 健康检查

如果你访问 `/api/health`：

1. 请求进入 [health.py](C:/Users/dlv20/Documents/合同合规审查/apps/api/app/routes/health.py)
2. 返回 `{\"status\": \"ok\"}`

这条链最重要的意义不是业务，而是确认服务活着。

## 4. 两个页面分别在表达什么

### 4.1 工作台页

- [dashboard.html](C:/Users/dlv20/Documents/合同合规审查/apps/api/app/templates/dashboard.html)

这个页面表达的是：

- 当前系统是什么
- 当前任务队列里有什么
- 这个产品的完整审查流程是什么

所以它更像一个“入口页 / 控制台页”。

### 4.2 审查结果页

- [review.html](C:/Users/dlv20/Documents/合同合规审查/apps/api/app/templates/review.html)

这个页面表达的是：

- 某一个合同任务的审查结果
- 哪些条款有问题
- 风险是什么
- 证据在哪里
- 制度依据是什么

这是整个 `MVP` 最关键的页面，因为客户最容易从这里感受到“系统真的能审合同”。

## 5. 为什么现在先用 sample_data

因为 `MVP` 的第一步不是先把所有后端接复杂，而是先把“结果长什么样”做出来。

当前用 `sample_data` 有 3 个直接好处：

1. 前端页面可以先稳定下来
2. 你可以先确认信息架构是否合理
3. 后面接真实链路时，只需要把数据源从“写死字典”替换成“真实服务结果”

这会明显减少前后端同时变动带来的混乱。

## 6. 下一步代码会怎么演进

当前代码最自然的演进顺序是：

1. 加上传接口和任务创建接口
2. 把任务数据从 `sample_data.py` 迁移到数据库或任务对象
3. 接入本地 `RAGFlow`
4. 接入 `MiniMax / Qwen`
5. 接入规则引擎
6. 让 `review.html` 展示真实的抽取结果和风险结果

也就是说，页面骨架本身不用推倒重来，后面主要是逐步把“假数据”替换成“真实链路”。
